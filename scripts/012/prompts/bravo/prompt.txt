# Flask Job Queue Application Specification

## Goal

You are to create a complete Python web application using the Flask framework and SQLAlchemy ORM.
The application will serve as a job queuing and processing platform.
The project must use SQLite as its database engine, and all user-uploaded text files (like configuration or results) must be stored directly in the database as Text.
The code should be well-structured using a Flask application factory pattern and organized with Blueprints.

The final deliverable should be the complete source code, including a requirements.txt file, and a README.md with clear instructions on how to set up the environment,
initialize the database, create an admin user, and run the application.

Step-by-Step Task List

## Database Models

Set up the Flask project structure and define the following SQLAlchemy models to represent the application's data.

### `User` Model
For human users who log in through the website.
- `id`: Primary Key
- `username`: String, unique, not nullable
- `password_hash`: String, not nullable
- `role`: String, either 'admin' or 'client'

### `Worker` Model
For the automated scripts. Workers do not log in with passwords.
- `id`: Primary Key
- `prefix`: String, not nullable (e.g., "formatter")
- `suffix`: String, not nullable (e.g., "alpha")
- `api_token`: String, unique, not nullable. This will be used for authentication.
- `profile_md`: Text
- `parameters_yaml_template`: Text
- `help_md`: Text

**Note**: The full worker identifier is a derived property and should not be stored in the database.
It is generated by combining the prefix and suffix in the format `/{prefix}/{suffix}`.
A unique constraint should be applied to the combination of `prefix` and `suffix`.

### `Request` Model
Represents a job submitted to a worker.
- `id`: Primary Key
- `requester_user_id`: Nullable Foreign Key to the `User` model (for jobs submitted by clients/admins).
- `requester_worker_id`: Nullable Foreign Key to the `Worker` model (for jobs submitted by other workers).
- `target_worker_id`: Foreign Key to the `Worker` model (the worker assigned to the job).
- `submitted_at`: DateTime
- `status`: String (e.g., 'pending', 'processing', 'completed', 'failed')
- `input_parameters_yaml`: Text

### `Response` Model
The result of a completed job.
- `id`: Primary Key
- `request_id`: Foreign Key to the `Request` model.
- `result_yaml`: Text
- `created_at`: DateTime

## Admin Interface (Admin Blueprint)

Create a web interface protected for admin users only. This blueprint is for managing the entire platform. The admin must be able to:

1.  **Manage Users**: Perform full CRUD (Create, Read, Update, Delete) operations on client user accounts.
2.  **Manage Workers**: Perform full CRUD operations on workers. When creating a new worker, the system must automatically generate a secure, unique API token for it. The token should be displayed to the admin upon creation.
3.  **View All Requests**: See a dashboard listing all requests in the system, regardless of who submitted them.
4.  **Reassign Jobs**: Have the ability to edit a 'pending' request and change its `target_worker_id` to a different worker.

## Worker-Facing API (API Blueprint)

Develop a RESTful API for worker scripts. All endpoints in this blueprint must be protected and require **token-based authentication**. The worker script will provide its token in the `Authorization` header as a Bearer Token (e.g., `Authorization: Bearer <worker_api_token>`).

- **`PUT /api/worker/profile`**: Allows an authenticated worker to upload and update its `profile.md`, `parameters_yaml_template`, and `help.md`.
- **`GET /api/jobs/next`**: Allows a worker to fetch the next 'pending' job from its queue. When a job is fetched, its status should change to 'processing'. A worker must not be assigned a new job if it already has one in the 'processing' state.
- **`POST /api/jobs/<request_id>/submit`**: Allows a worker to submit the `results.yaml` for a job it was processing. This updates the request's status to 'completed' and creates the corresponding `Response` entry.
- **Worker-to-Worker Endpoints**:
    - **`POST /api/workers/<prefix>/<suffix>/request`**: Allows an authenticated worker to submit a job request to another worker, provided the target has a different `prefix`.
    - **`GET /api/requests/<request_id>/status`**: Allows a worker to check the status of a job it submitted.
    - **`GET /api/requests/<request_id>/result`**: Allows a worker to download the `results.yaml` from a completed job it submitted.

## Client-Facing Web Pages (Main Blueprint)

This blueprint contains the views and templates for logged-in 'client' and 'admin' users.

1.  **Authentication**: Implement login, logout, and registration pages for client users using Flask-Login.
2.  **Worker List Page**: A page that lists all workers and their key information. For each worker, display:
    - Its full identifier (e.g., `/formatter/alpha`).
    - Links to view their `profile.md`, `parameters_yaml_template`, and `help.md`.
    - Key stats: number of pending requests, average job completion time, and the timestamp of the last completed job.
3.  **Request Submission Page**: A form where a user can upload a `parameters.yaml` file and specify a worker to run the job.
    - The user can provide a full worker identifier (e.g., `/formatter/alpha`). The application should parse this string to find the corresponding worker by its `prefix` and `suffix`.
    - Alternatively, the user can provide just a `prefix` (e.g., `formatter`). In this case, the application logic must **assign the request to the worker with that prefix that currently has the shortest queue** of 'pending' requests.
4.  **User Dashboard Page**: A page where a user can view the status of all requests they have submitted and download the `results.yaml` for any completed jobs.
